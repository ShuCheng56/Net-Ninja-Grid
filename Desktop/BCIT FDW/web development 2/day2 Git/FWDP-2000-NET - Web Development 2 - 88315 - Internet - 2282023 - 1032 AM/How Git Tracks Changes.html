How Git Tracks Changes

Once you've initialized a project for Git, it can scan for new and edited files from that directory downwards and maintains an internal database (repository) of changes to those files.

To have it actually start tracking changes for any given file(s) in your project, you need to first add them to the repository.

Once added, you create point-in-time versions of those files by issuing the commit command along with a message indicating from a high level what has changed in that revision.  

You can think of commits as a sort of snapshot of the state of your project at a point in time - somewhat like if you were manually tracking versions of your code by numbering them v1, v2, v3, etc.

Note that, unlike some other version control software, Git does not track folders themselves - just the files therein.  For Git to be aware of a folder, you need to have at least one file within it.
